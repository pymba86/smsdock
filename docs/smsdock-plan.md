# smsdock

## Назначение

`smsdock` это monorepo-система для работы с несколькими GSM-модемами через AT-команды:

- прием входящих SMS;
- сохранение SMS в `SQLite`;
- просмотр SMS и статусов модемов в веб-интерфейсе;
- ручной поиск сети;
- ручной перевыбор оператора;
- работа в Docker внутри Linux VM на Proxmox.

## Технологии

- monorepo: `pnpm`;
- backend: Go;
- frontend: `React + Vite + lucide-react`;
- база данных: `SQLite`;
- UI-обновление: polling открытых экранов + кнопка `Обновить`;
- realtime через `SSE` не используется;
- SIM PIN не используется.

## Ключевые правила

- модемов несколько;
- подключение модемов идет через Linux serial-устройства;
- идентификатор модема в системе: `IMEI`;
- `/dev/ttyUSB*` не считается постоянным идентификатором;
- конфигурации устройств в приложении нет;
- добавление модемов выполняется только через UI;
- сканирование доступных устройств выполняется только в момент добавления модема;
- выбор сети только вручную;
- автоматическое переключение на другого оператора запрещено;
- `runtime_device_path` в БД не хранится;
- после рестарта или потери устройства backend должен заново найти уже зарегистрированный модем по `IMEI`.

## Backend

Основные модули `packages/api`:

- `serial`: открытие и переоткрытие serial-портов;
- `at`: очередь AT-команд, таймауты, защита от зависших сессий;
- `modem`: runtime-состояние модема и rebind по `IMEI`;
- `sms`: polling SMS, декодирование, дедупликация, сохранение;
- `network`: поиск сетей и ручной выбор сети;
- `storage`: `SQLite`, миграции, репозитории;
- `http`: REST API;
- `health`: `/health` и `/ready`;
- `fakemodem`: эмулятор для тестов и CI.

Один зарегистрированный модем соответствует одному worker.

Для одного модема одновременно не выполняются:

- polling SMS;
- поиск сети;
- выбор сети;
- recovery.

## Frontend

Основные экраны `packages/web`:

- список модемов со статусами;
- экран SMS конкретного модема;
- экран добавления модема через scan;
- карточка модема с поиском сети, ручным выбором сети и журналом событий.

## Обновление UI

- `SSE` не используется;
- экран модемов обновляется раз в `10-15 сек`;
- экран SMS открытого модема обновляется раз в `10 сек`;
- при неактивной вкладке polling можно останавливать;
- кнопка `Обновить` остается в интерфейсе всегда.

## Прием и хранение SMS

- прием SMS выполняется только через периодический опрос модема;
- прием выполняется в `PDU mode`;
- в первой версии поддерживаются `GSM 7-bit`, `UCS2` и multipart через `UDH`;
- после успешного сохранения SMS удаляется из памяти модема;
- в `SQLite` SMS хранятся бессрочно;
- дедупликация обязательна;
- в API и UI сразу закладываются пагинация и сортировка.

Минимальный dedupe-ключ должен учитывать:

- `modem_id`;
- отправителя;
- время сообщения;
- `raw_pdu` или его хеш.

## События и журнал

Логировать нужно:

- добавление, включение и выключение модема;
- потерю связи с модемом;
- recovery;
- потерю сети;
- поиск сети;
- ручной выбор сети;
- ошибки чтения и декодирования SMS;
- смену статусов модема.

Не логировать:

- каждый успешный poll;
- каждую успешную AT-команду;
- heartbeat без изменений.

Рекомендация по хранению событий:

- `info`: `14 дней`;
- `warn` и `error`: `90 дней`.

## Работа с сетью

- поиск сети запускается только вручную из UI;
- поиск сети выполняется только для выбранного модема;
- во время поиска сети polling SMS для этого модема ставится на паузу;
- пользователь выбирает сеть вручную;
- в БД сохраняется `assigned_network_mcc_mnc`;
- после сбоев backend восстанавливает регистрацию только в закрепленной сети;
- самостоятельный переход на другого оператора запрещен.

## Надежность и эксплуатация

В базовый scope входят:

- автопереподключение к модему;
- переоткрытие serial-порта при ошибках;
- таймаут на каждую AT-команду;
- retry с ограничением и `backoff`;
- защита от зависших AT-сессий;
- статусы `ready`, `polling`, `scanning`, `recovering`, `degraded`, `offline`;
- история ошибок и событий в `SQLite`;
- `health` и `ready` endpoint;
- fake modem для CI и интеграционных тестов.

Recovery pipeline:

1. повтор команды;
2. переоткрытие порта;
3. переинициализация AT-сессии;
4. проверка SIM, сигнала и регистрации;
5. повторная регистрация только в закрепленной сети;
6. при неудаче перевод модема в `degraded` или `offline`.

## Docker и Proxmox

- USB-модемы пробрасываются из Proxmox в Linux VM;
- в VM лучше использовать стабильные device names через `/dev/serial/by-id/...` или `udev`-алиасы;
- Docker получает доступ к ним через `devices`;
- приложение работает только с теми устройствами, которые уже видит контейнер;
- новый физический модем сначала должен стать доступен VM и контейнеру, после этого его можно добавить через UI;
- для `SQLite` обязателен persistent volume.

## Этапы реализации

1. Поднять `pnpm` monorepo и каркас `packages/api`, `packages/web`, `packages/shared`.
2. Сделать `SQLite`, миграции и базовые таблицы.
3. Реализовать serial/AT-слой и fake modem.
4. Реализовать добавление модема через UI и регистрацию по `IMEI`.
5. Реализовать polling SMS, `PDU`, `UCS2` и multipart.
6. Сделать экран модемов, экран SMS и журнал событий.
7. Реализовать ручной поиск сети и ручной выбор оператора.
8. Добавить recovery, `health`/`ready` и эксплуатационные статусы.
9. Подготовить Docker Compose и сценарий запуска в Proxmox VM.
10. Закрыть интеграционные тесты и отказоустойчивые сценарии.
